\section{MODELAGEM UML}

Esta seção apresenta a modelagem visual do sistema \textbf{MedAgenda} utilizando a Linguagem de Modelagem Unificada (UML), em conformidade com as especificações canônicas de Booch, Rumbaugh e Jacobson \cite{booch2005}. Os diagramas foram projetados com rigor para cobrir os aspectos funcionais (casos de uso), estruturais (diagrama de classes e componentes) e comportamentais (diagramas de sequência e de estados do objeto) do sistema, mantendo estrita rastreabilidade com os requisitos e regras de negócio detalhadas na Seção 2 \cite{larman2004}.

\subsection{Diagrama de Casos de Uso}

O Diagrama de Casos de Uso delimita a fronteira do sistema com o mundo externo, mapeando as interações transacionais entre os atores corporativos e os serviços fundamentais providos pela plataforma \cite{booch2005}. Foram identificados e modelados três atores primários no domínio médico:
\begin{itemize}
    \item \textbf{Paciente}: Ator que interage no agendamento, cancelamento, busca por profissionais de saúde e acompanhamento do seu histórico de atendimentos.
    \item \textbf{Médico}: Ator que gerencia suas faixas horárias de trabalho, consulta a agenda diária e realiza o registro de prontuários clínicos e diagnósticos.
    \item \textbf{Administrador}: Ator responsável pela administração geral da clínica, gerência de cadastros do corpo médico e emissão de relatórios consolidados de gestão.
\end{itemize}

\subsection{Diagrama de Classes}

O Diagrama de Classes descreve a espinha dorsal estrutural e o modelo conceitual do MedAgenda, formalizando as entidades transacionais, seus atributos tipados, operações semânticas de mutação de estado e os relacionamentos e cardinalidades do domínio \cite{larman2004, meyer1997}:

\begin{itemize}
    \item \textbf{Usuario (Classe Abstrata)}
    \begin{itemize}
        \item \textit{Atributos}: \texttt{- id: int}, \texttt{- nome: String}, \texttt{- email: String}, \texttt{- senhaHash: String}
        \item \textit{Métodos}: \texttt{+ login(senha: String): bool}, \texttt{+ logout(): void}, \texttt{+ obterPainelInicial(): String}
    \end{itemize}
    \item \textbf{Paciente (Especialização de Usuario)}
    \begin{itemize}
        \item \textit{Atributos}: \texttt{- cpf: String}, \texttt{- dataNascimento: Date}, \texttt{- telefone: String}, \texttt{- endereco: String}
        \item \textit{Métodos}: \texttt{+ agendarConsulta(m: Medico, dt: DateTime): Consulta}, \newline \texttt{+ cancelarConsulta(idConsulta: int): bool}
    \end{itemize}
    \item \textbf{Medico (Especialização de Usuario)}
    \begin{itemize}
        \item \textit{Atributos}: \texttt{- crm: String}, \texttt{- especialidade: String}, \texttt{- valorConsulta: float}
        \item \textit{Métodos}: \texttt{+ gerenciarHorarios(h: HorarioDisponivel): void}, \newline \texttt{+ registrarProntuario(c: Consulta, anamnese: String): Prontuario}
    \end{itemize}
    \item \textbf{Administrador (Especialização de Usuario)}
    \begin{itemize}
        \item \textit{Atributos}: \texttt{- nivelAcesso: int}
        \item \textit{Métodos}: \texttt{+ cadastrarMedico(m: Medico): bool}, \newline \texttt{+ gerarRelatorio(tipo: String, p: Periodo): Relatorio}
    \end{itemize}
    \item \textbf{Consulta (Entidade Central de Negócio)}
    \begin{itemize}
        \item \textit{Atributos}: \texttt{- id: int}, \texttt{- dataHora: DateTime}, \texttt{- status: String}, \texttt{- observacao: String}
        \item \textit{Métodos}: \texttt{+ confirmar(): void}, \texttt{+ cancelar(motivo: String): void}, \texttt{+ concluir(): void}
    \end{itemize}
    \item \textbf{Prontuario (Composição de Consulta)}
    \begin{itemize}
        \item \textit{Atributos}: \texttt{- id: int}, \texttt{- anamnese: String}, \texttt{- diagnostico: String}, \texttt{- prescricao: String}
        \item \textit{Métodos}: \texttt{+ assinarDigitalmente(): bool}, \texttt{+ obterResumoClinico(): String}
    \end{itemize}
    \item \textbf{HorarioDisponivel (Entidade de Agendamento)}
    \begin{itemize}
        \item \textit{Atributos}: \texttt{- id: int}, \texttt{- diaSemana: int}, \texttt{- horaInicio: String}, \texttt{- ativo: bool}
        \item \textit{Métodos}: \texttt{+ bloquearHorario(): void}, \texttt{+ liberarHorario(): void}
    \end{itemize}
\end{itemize}

\textbf{Relacionamentos Estruturais e Multiplicidades:}
\begin{itemize}
    \item \textbf{Paciente (1) para (0..* ) Consulta}: Associação onde o paciente atua como titular de múltiplos agendamentos no seu ciclo de vida na clínica.
    \item \textbf{Medico (1) para (0..* ) Consulta}: Associação que vincula o profissional responsável por atender cada consulta médica agendada.
    \item \textbf{Consulta (1) para (0..1) Prontuario}: Relação de \textbf{Composição forte}, onde o registro de prontuário eletrônico é subordinado diretamente à existência de uma consulta médica previamente concluída \cite{larman2004}.
    \item \textbf{Medico (1) para (1..* ) HorarioDisponivel}: Associação que separa a grade horária em uma entidade autônoma para permitir pesquisas eficientes no banco de dados.
\end{itemize}

\subsection{Diagramas de Sequência}

Os Diagramas de Sequência demonstram a ordem cronológica e a troca de mensagens transacionais entre os controladores, serviços e repositórios para concretizar os fluxos críticos de negócio \cite{booch2005}:

\subsubsection{DS-01: Agendamento de Consulta (Atende ao RF-03)}
\begin{enumerate}
    \item O \textbf{Paciente} seleciona a especialidade, médico, data e horário diretamente na interface web.
    \item A camada de apresentação envia a requisição para o \texttt{ConsultaController}.
    \item O controlador consulta o \texttt{HorarioService} para confirmar se a faixa horária solicitada ainda está ativa e não foi reservada por outra transação concorrente.
    \item Com a validação positiva, o \texttt{ConsultaController} instancia uma nova \texttt{Consulta} com status inicial \texttt{AGENDADA}.
    \item O \texttt{ConsultaRepository} executa a inserção transacional do registro no banco de dados relacional.
    \item O \texttt{NotificacaoService} é notificado em background para encaminhar o e-mail de confirmação ao paciente.
\end{enumerate}

\subsubsection{DS-02: Cancelamento de Consulta (Atende ao RF-04)}
\begin{enumerate}
    \item O \textbf{Paciente} solicita o cancelamento na sua lista de agendamentos, informando o motivo.
    \item O \texttt{ConsultaController} delega ao \texttt{ConsultaService} a verificação temporal do agendamento em comparação com o relógio do servidor.
    \item Caso a diferença entre o horário atual e a data da consulta seja superior a 24 horas, a regra de negócio do RF-04 autoriza a transição do status para \texttt{CANCELADA}.
    \item O \texttt{HorarioService} é acionado para reativar o registro de \texttt{HorarioDisponivel}, disponibilizando o slot para outros pacientes da clínica.
    \item O \texttt{NotificacaoService} emite comunicados automáticos de cancelamento tanto para o paciente quanto para o médico responsável.
\end{enumerate}

\subsubsection{DS-03: Autenticação no Sistema (Atende ao RF-02)}
\begin{enumerate}
    \item O usuário submete suas credenciais (e-mail e senha) no formulário da tela inicial.
    \item O \texttt{AuthService} processa a requisição consultando o \texttt{UsuarioRepository}.
    \item O serviço executa a comparação segura do hash da senha digitada contra o registro criptografado persistido.
    \item Confirmada a autenticidade, o servidor gera um token de sessão com as permissões de acesso e redireciona o usuário para seu respectivo painel interativo.
\end{enumerate}

\subsection{Diagramas de Gráficos de Estado}

A modelagem dinâmica de estados mapeia todas as transições legais que as entidades centrais podem assumir ao longo do processamento transacional do sistema \cite{booch2005}:

\subsubsection{1. Ciclo de Vida do Objeto \texttt{Consulta}}
\begin{itemize}
    \item \textbf{AGENDADA}: Estado inicial, atribuído no momento em que o paciente conclui a solicitação de agendamento no sistema.
    \item \textbf{CONFIRMADA}: Estado atingido após a verificação automática pela secretaria ou confirmação prévia via notificação.
    \item \textbf{CANCELADA}: Estado final alternativo, acionado quando o paciente ou a clínica desmarcam a consulta respeitando a antecedência mínima de 24 horas.
    \item \textbf{EM\_ANDAMENTO}: Transição acionada no momento em que o paciente entra no consultório e o médico inicia o atendimento na tela diária.
    \item \textbf{CONCLUIDA}: Estado final definitivo, estabelecido obrigatoriamente logo após o preenchimento integral dos dados clínicos e assinatura digital do prontuário pelo profissional.
\end{itemize}

\subsubsection{2. Ciclo de Vida do Objeto \texttt{Paciente}}
\begin{itemize}
    \item \textbf{INATIVO}: Estado provisório do cadastro durante o período de verificação do endereço de e-mail no ato da criação da conta.
    \item \textbf{ATIVO}: Estado operacional padrão, no qual o paciente possui permissão plena para agendar e cancelar consultas no sistema.
    \item \textbf{BLOQUEADO}: Estado de exceção acionado automaticamente ou por administradores em casos de excesso de ausências injustificadas (no-show), impedindo novos agendamentos online sem prévio contato com a clínica.
\end{itemize}

\subsection{Diagrama de Componentes e Implantação}

A arquitetura lógica de componentes ilustra a decomposição modular do sistema e suas conexões por meio de interfaces bem definidas e desacopladas \cite{fowler2002}:
\begin{itemize}
    \item \textbf{Camada de Apresentação (Frontend React)}: Componentes visuais (\texttt{AgendamentoView}, \texttt{ProntuarioView}, \texttt{DashboardView}) que interagem com o usuário e se comunicam exclusivamente por chamadas HTTP RESTful com o backend.
    \item \textbf{Camada de Aplicação (API REST Node.js/Express)}: Componentes transacionadores e de domínio (\texttt{ConsultaController}, \texttt{ConsultaService}, \texttt{NotificacaoService}, \texttt{AuthService}) responsáveis pela validação das regras de negócio, autenticação JWT e controle do fluxo de dados \cite{martin2019}.
    \item \textbf{Camada de Persistência (Prisma ORM $\times$ PostgreSQL)}: Componentes de acesso a dados (\texttt{ConsultaRepository}, \texttt{UsuarioRepository}) que traduzem as chamadas orientadas a objetos da aplicação para consultas relacionais seguras, garantindo conformidade transacional ACID sobre as tabelas físicas do banco de dados \cite{silberschatz2020}.
\end{itemize}
